[RECONNAISSANCE]
----------------
- GET / -> 302 redirect to /login
- GET /login -> 200, login form with username/password fields
- POST /login (bad creds) -> 401, "Username/Password Combination is invalid" (Content-Length: 4404)
- Directory fuzzing (common.txt, directory-list-2.3-medium.txt, POST methods) -> no routes found except /login
- JS on login page checks cookie "token" client-side only (cosmetic modal display)
- After auth: GET / returns dashboard with student table, each row has data-id (Base64 encoded)
- JS file /assets/js/app.min.js loaded post-auth: activates /update-student/<id> links only if staff.admin === true

[VULNERABILITY DISCOVERY]
-------------------------
1. SQL Injection (Auth Bypass): login form vulnerable to classic OR-based SQLi
   - No sanitization on username/password fields
   - MySQL backend (confirmed by # comment behavior)

2. IDOR (Insecure Direct Object Reference): /update-student/<base64_id> accessible without ownership check
   - Logged in as rhonda.daniels, accessed Natasha Drew's record directly
   - No server-side validation that the student belongs to the logged-in teacher

3. Client-Side Access Control Bypass: staff.admin = false in JS prevents button click
   - Server does NOT enforce this check on POST /update-student/<id>
   - Direct POST bypasses JS restriction entirely

[LESSONS LEARNED]
-----------------
- Never rely on client-side JS checks for access control (admin flags, button states)
- IDOR: always verify server-side that the requesting user owns/has rights to the resource
- SQLi: parameterized queries / prepared statements prevent OR-based auth bypass
- Base64 is encoding, not encryption - never use it as a security mechanism for IDs
- Student IDs followed predictable pattern (Firstname_Lastname) making enumeration trivial
